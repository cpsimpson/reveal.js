Technical Documentation

  • 3 main kinds

    • Step-by-step Tutorials
    • Topical Guides
    • Reference Materials

  • Additional kinds

    • FAQ
    • Readme files

Tutorials

Quick

  • Success in 30 minutes
  • Highlight when bad decisions are made for brevity

Easy

  • Test in different scenarios
  • Guarantee success for the user

Not too easy

  • Ensure that users that won't succeed with your project don't breeze through the tutorial

Reflect the project feel

  • Demonstrate a wide cross-cut of features in the tutorial

Topical Guides

Deep dive into an area of functionality

  • Doesn't need to have every detail
  • Provides the "why"

Comprehensive

  • User comfortable with the topic after a close read

Skipped by a lot of projects

  • Rely on books

Reference Materials

Fully document public APIs

Don't just rely on autodocs

Describe the parameters and their behavioural influence

Provides the "how"

Frequently Asked Questions

  • Better if you can fix the cause of the question
  • Important to keep them up-to-date

Readme Files

  • Often first introduction to a project
  • Important for projects stored in public repositories
  • Capture important details
  • Provide links to more information

Structure

Have some

  • Define the structure of your documenation up front
  • Organic structure hard to follow
  • Adjust as needed
  • Do so knowingly/thoughtfully

Timely Updates

Update your documentation when the code changes

Writing Good Docs

Write more

  • Have an editor
  • Write more
  • Read more
  • Analyze what you are reading for effectiveness
  • Write more
  • Follow a style-guide, even if loosely
  • Be consistent in your style
  • Write more

Style

Be conversational

  • Strike a personal tone (Use I and we)
  • Helps readers identify with the material

Be careful with tense

  • Usually second person, future tense
  • "Then, when you've opened the menu, click 'Settings'"

Be active rather than passive

Be concise

Omit the fluff

  • "A good tutorial should show off most of the different areas of the project."
  • "A tutorial should show off the major project areas."

Print vs Screen

People skim online, read in print

Use markup online

  • Break up sections with headers
  • Use structural elements - lists, tables, diagrams
  • Use bold and italics more liberally
  • Shorter paragraphs

Editing Tips

Don't edit without permission

  • You'll probably sound like a jerk

Edit on paper if possible

Have references available

  • Style guide
  • Dictionary
  • Thesaurus

Read slowly

  • Try reading out loud

Make multiple passes

Be consistent

Self-Editing Tips

  • Separate editing from writing
  • Leave some time between writing and editing
  • Try changing the margins
  • Try changing locations

Sphinx

Generate beautiful documentation from plain text

Focus on hand-crafted documenation, possibly combining with autodocs

History

  • Created for Python's documentation
  • Publicly released in 2008
  • Now used by many projects

Format

Uses reStructuredText

plain-text markup

Can be easily stored in source control

Output to Many Formats

  • HTML
  • PDF
  • ePub
  • LaTeX
  • man pages
  • ...

docutils

reStructuredText

  • Markup often makes text ugly, hard to read
  • Readable and simple
  • Powerful enough for non-trivial uses
  • Similar to, but more powerful and extensible than, markdown
